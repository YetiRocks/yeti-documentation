    SDK Overview

    The Yeti SDK (yeti-sdk) is the developer-facing API for building Yeti applications. It is built on yeti-types (which defines the core Resource trait and Context struct) and re-exports everything you need through a single import:

    use yeti_sdk::prelude::*;

    What the prelude provides

    CategoryKey exports
    Resource Typesresource!, extends_table!, register_resource!, Resource, Context, ContextExt
    Response Buildersok(), reply(), created(), no_content(), not_found(), json_response(), html_response(), text_response()
    Request ParsingRequestBodyExt, RequestExt, ContextExt::require_json_body(), ContextExt::require_id()
    Table AccessTable, Tables, TableExt, KvBackend, BackendManager
    Query & IndexingQueryBuilder, QueryExt, FiqlQuery, Condition, Comparator
    AuthAccessControl, AuthIdentity, AuthProvider, AuthHook, CookieJar
    Encoding & Cryptobase64_encode/decode, base64url_encode/decode, sha256, sha512, hmac_sha256, hex_encode/decode, url_encode/decode, KeyEncoder
    Filesystemread_file, write_file, file_exists, mkdir, read_dir, append_file, copy_file, remove_file, remove_dir (root-directory-aware)
    HTTP ClientFetchBuilder, FetchResponse, fetch! macro
    Real-timePubSubManager, subscribe, publish, connect methods on Resource
    Loggingyeti_log! macro (dylib-safe, bridges to host tracing)

    The prelude also re-exports commonly used external types: Request, Response, StatusCode, Method, json!, Value, and HashMap. Error types (YetiError, ProblemDetails, BadRequest, Forbidden, NotFoundError, Unauthorized) and registration types (AppRegistration) are also included.

    Application architecture

    Yeti applications compile to dynamic libraries (.dylib). The compiler reads your config.yaml, copies resource .rs files into a build cache, generates a lib.rs that wires everything together, and compiles the result. You write individual resource files; the toolchain handles the rest.

    Each resource file defines a struct that implements the Resource trait (usually via the resource! macro). The struct name becomes the route path (lowercased by default).

    Minimal example

    use yeti_sdk::prelude::*;

resource!(Hello {
    get => json!({"message": "Hello, World!"})
});

    This registers a resource at GET /my-app/hello. The resource! macro generates the struct, implements the Resource trait, and registers it for export automatically.

    Context-based example

    The Context struct carries everything about a request: method, path, body, headers, auth identity, query parameters, path ID, and access to the application's tables.

    use yeti_sdk::prelude::*;

resource!(Items {
    get(ctx) => {
        let table = ctx.get_table("Items")?;
        match ctx.path_id.as_deref() {
            Some(id) => {
                let item = table.get(id).await?;
                ok(item)
            },
            None => {
                let all = table.get_all().await?;
                ok(json!(all))
            }
        }
    },
    post(ctx) => {
        let body = ctx.require_json_body()?.clone();
        let table = ctx.get_table("Items")?;
        let id = body["id"].as_str().unwrap_or("unknown");
        table.put(id, body.clone()).await?;
        created(body)
    },
    delete(ctx) => {
        let id = ctx.require_id()?;
        let table = ctx.get_table("Items")?;
        table.delete(id).await?;
        no_content()
    }
});

    Dylib constraints

    Code running inside applications executes in a dynamically loaded library. This imposes several constraints:

    • No tracing::info! -- use yeti_log! instead (tracing TLS is isolated per dylib)
    • No tokio::spawn -- causes crashes. Use futures::stream::unfold for async patterns
    • No reqwest::blocking::Client -- crashes due to internal tokio conflict. Use fetch! for HTTP
    • No host statics -- OnceLock values in the host binary are duplicated in dylib memory
